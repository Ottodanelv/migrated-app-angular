## Overview

This document defines the rules, conventions, and patterns for migrating **gestionWeb** (Cetelem financial token management webapp) from **Spring MVC JSP** to **Angular 22 standalone**.

**All MetaAgents** (MetaPlanner, MetaTechPlanner, MetaCoder, MetaReviewer) **must** follow these rules. Deviations require explicit approval.

## Migration Rules: DO

These are **required** patterns. MetaCoder must implement every module following these rules:

1. **Use `inject()` for DI**: Always use the `inject()` function, never constructor injection with `private readonly`.
   ```typescript
   private readonly http = inject(HttpClient);
   ```

2. **Use Signals for state**: All reactive state must use `signal()` / `computed()` / `effect()`. No plain variables for component state.
   ```typescript
   protected readonly infoOperacion = signal<InfoOperacion | null>(null);
   protected readonly loading = signal(false);
   ```

3. **Use `@if`, `@for`, `@defer` control flow**: No `*ngIf`, `*ngFor`, or `*ngSwitch`.
   ```html
   @if (loading()) {
     <div class="spinner">Loading...</div>
   } @for (item of items(); track item.id) {
     <div>{{ item.name }}</div>
   }
   ```

4. **Standalone components only**: Every component must be `standalone: true` (default in Angular 22). No `NgModule` declarations.

5. **Use Angular HttpClient**: All HTTP calls must use `HttpClient` with typed responses. No `fetch()`, no jQuery AJAX.
   ```typescript
   private readonly http = inject(HttpClient);
   readonly operacion$ = this.http.get<Operacion>('/api/operacion');
   ```

6. **Functional guards for routing**: Use `CanActivateFn`, not guard classes.
   ```typescript
   export const authGuard: CanActivateFn = () => {
     const auth = inject(AuthService);
     return auth.isAuthenticated();
   };
   ```

7. **Tailwind CSS utility classes**: All styling must use Tailwind utility classes. No custom CSS in component styles unless absolutely needed (use `@apply` in the global CSS file).

8. **Multi-brand via CSS variables + data attribute**: Use `[data-sociedad="800"]` variant to theme per brand.
   ```css
   @custom-variant sociedad-800 (&:where([data-sociedad="800"], [data-sociedad="800"] *));
   ```

9. **Reactive Forms for forms**: Use `FormGroup` + `FormControl` + `Validators`. No template-driven forms.

10. **TypeScript strict mode**: Use `strict: true` in `tsconfig.json`. Define interfaces for all data models.

11. **One component per file**: Each component, service, guard, interceptor in its own file.

12. **Use `provideRouter()` + `withComponentInputBinding()`**: Configure routes with input binding for clean parameter access.

13. **HttpInterceptor for auth + error handling**: Single interceptor for adding auth tokens and handling errors globally.

14. **Test every component**: Every component must have at least one smoke test (renders without error).

15. **Use `REMOVED` marker**: When a source concept has no equivalent in the destination, mark it explicitly in comments.

## Migration Rules: DON'T

These patterns from the source stack are **forbidden**. MetaReviewer must flag any PR containing these:

1. **No JSP scriptlets** (`<% %>`, `<%= %>`, `${}` expressions): These are server-side patterns. Use Angular template syntax.

2. **No JSTL tags**: No `<c:if>`, `<c:forEach>`, `<c:choose>`, `<fmt:formatNumber>`, `<spring:message>`. Use Angular control flow and i18n.

3. **No jQuery**: No `$.ajax`, `$().val()`, `$().on()`, or any jQuery API. Use Angular HttpClient and template bindings.

4. **No Bootstrap CSS classes**: No `.col-sm-*`, `.row`, `.container`, `.btn`, `.form-group`. Use Tailwind equivalents.

5. **No `HttpSession` references**: No `session.getAttribute()`, `session.setAttribute()`. State is client-side with Signals.

6. **No field injection in classes**: No `@Autowired` or Angular's `private readonly service: Service` in constructor. Use `inject()`.

7. **No `NgModules`**: No `@NgModule({...})` declarations. Angular 22 is standalone-only by default.

8. **No `*ngIf`, `*ngFor`, `*ngSwitch`**: These are deprecated in favor of `@if`, `@for`, `@switch`.

9. **No server-side view resolution**: No `ModelAndView`, no `InternalResourceViewResolver`. Use Angular Router.

10. **No `any` types**: All TypeScript types must be explicit interfaces or types. No `any` unless with explicit documented exception.

11. **No direct DOM manipulation**: No `document.getElementById()`, `document.querySelector()` or imperative DOM access. Use Angular template refs.

12. **No SOAP/WS client logic in frontend**: The legacy had SOAP clients; the Angular app must call REST endpoints. If the backend is not being migrated, wrap SOAP calls in a new REST API layer.

13. **No duplicated brand CSS**: No separate CSS files per brand. Use Tailwind CSS variables + variants.

14. **No `private` methods in templates**: Methods called from templates must be `protected` or `public`.

15. **No unused imports**: Remove all unused imports. TypeScript strict mode will catch unused declarations.

## Testing Strategy

### Unit Tests

- **Framework**: Vitest (configured in workspace)
- **Runner**: `ng test` (Vitest via Angular CLI)
- **Pattern**: `*.spec.ts` co-located with source files
- **Setup**: `describe()` / `it()` / `expect()` (Vitest API)
- **Mocking**: `vi.mock()` for services, `vi.fn()` for callbacks
```typescript
import { describe, it, expect, vi, beforeEach } from 'vitest';
import { GestionTokenService } from './gestion-token.service';

describe('GestionTokenService', () => {
  it('should return operacion data', async () => {
    const service = new GestionTokenService(mockHttp);
    const result = await service.validarToken('test-token');
    expect(result).toBeDefined();
  });
});
```

### Component Tests

- **Setup**: `render(Component, { providers: [...] })` from `@testing-library/angular` or Vitest browser mode
- **Mock HTTP**: Use MSW (Mock Service Worker) with `setupServer`
- **Verify**: Template output, signal states, event handlers
```typescript
import { render, screen } from '@testing-library/angular';

it('should display loading state', async () => {
  await render(InfoOperacionComponent, {
    providers: [provideHttpClient(withFetch())],
  });
  expect(screen.getByText('Cargando...')).toBeDefined();
});
```

### Integration Tests

- **HTTP mocking**: MSW handlers for all API endpoints
- **Route testing**: Test navigation between pages via Router
- **Brand theming**: Test with different `data-sociedad` values

## Multi-Brand Strategy (Sociedades 400, 600, 800)

The legacy app supports 3 brands via `sociedad` model attribute. In Angular:

1. **CSS Variables per brand** in `styles.css`:
```css
@import "tailwindcss";

@theme {
  --color-brand-primary: oklch(0.45 0.12 260); /* Default */
  --color-brand-secondary: oklch(0.6 0.15 280);
  /* ... shared tokens */
}

/* Sociedad 800 - Cajamar (primary today) */
@custom-variant sociedad-800 (&:where([data-sociedad="800"], [data-sociedad="800"] *));

/* Sociedad 600 - Xfera */
@custom-variant sociedad-600 (&:where([data-sociedad="600"], [data-sociedad="600"] *));

/* Sociedad 400 - Default */
@custom-variant sociedad-400 (&:where([data-sociedad="400"], [data-sociedad="400"] *));
```

2. **Dynamic brand switching**: A service manages `data-sociedad` on `<body>` or root element.
3. **Component templates**: Components use the current brand when needed via a `sociedad` signal.
